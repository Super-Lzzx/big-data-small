# 如何对比并验证新线程互补性公式
系统的对比实验。下面给出对比思路、可选的 baseline、评价指标，以及实验流程建议。

---

## 一、为什么要做对比

1. **验证有效性**  
   - 通过对比实验，展示：在相同的多线程/多核场景下，你的新公式输出的互补度分数与实际性能（吞吐量、IPC、迁移后提升等）之间的相关性更高、误差更小。  
   - 若无对比，读者很难判断“这个新公式到底有没有比前人好”。

2. **凸显创新点**  
   - 现有文献中常见的方法往往只考虑几种资源（如只看 L1/L2 miss 差异，或只关注 IPC 差异）。你要证明：  
     1. “同时纳入更多维度”（L1 miss、L2 miss、iTLB miss、dTLB miss、分支未命中、IPC 等）  
     2. “加权／归一化方式更合理”  
   - 并通过实验数据说明，这些新增或改进能更好地反映线程间真实的资源竞争与互补关系。

3. **证明鲁棒性**  
   - 在多种不同负载组合（CPU 密集、内存密集、I/O 密集、混合型、生产型等）下，你要展示：  
     - 新公式在各类场景中均保持较高准确度  
     - 误差小并且稳定，不像某些基线方法只在特定场景有效

## 二、可选的“对比对象”（Baseline 方法）

以下是常见、且有代表性的几类两线程互补度度量方式，可作为 baseline，与你的新公式做对照：

1. **只看 IPC 差值（Efficiency Gap）**  
    C_IPC(A,B) = 1 / (1 + |IPC_A - IPC_B|)
   或者直接用 |IPC_A - IPC_B|  当作互补度分数（数值小表示互补好／冲突小）。

2. **只看某一层缓存竞争（如 L2 miss 差值）**  
   C_L2(A,B) = 1 / (1 + |L2miss_A - L2miss_B|)
   或直接用 |L2miss_A - L2miss_B| 作为负分数。

3. **加权和（文献中常见）**  
   - 有些论文把几种硬件事件的差值用加权求和，再取倒数作为互补度：  
     Overlap(A,B) = w1 × |L1miss_A - L1miss_B| 
              + w2 × |L2miss_A - L2miss_B| 
              + w3 × |IPC_A - IPC_B|
    C_weighted(A,B) = 1 / (1 + Overlap(A,B))
   - 如果文献中没有公开权重，就以等权或文中给的示例为准做复现。

4. **基于方差/极差的度量**  
   - 有的工作直接把两条线程在某项资源（如 L1/L2 miss、IPC 等）上组成向量 \(\{r_A, r_B\}\)，计算方差或极差：  
     Var(r) = Var(r_A, r_B)
      Range(r) = max(r_A, r_B) - min(r_A, r_B)
   - 再将所有资源维度上的方差之和或极差之和带入公式：  
     C_var(A,B) = 1 / (1 + sum_{r in R} Var(r) + Var(IPC))

## 三、度量指标（如何量化“好/坏”）

在对比实验中，需要选择合理的评价指标来衡量“预测分数” 与 “真实性能” 之间的吻合程度。常见指标包括：

1. **相关系数（Pearson / Spearman Correlation）**  
   - 把真实的同核性能度量（如两线程并行时的 IPC_sum、执行时间、吞吐量等）视为“标签” \(y_{\text{true}}\)。  
   - 把各方法算出的互补度分数 \(\hat y(A,B)\) 视为“预测值”。  
   - 计算 Pearson 或 Spearman 相关系数：相关系数越高，说明分数与真实性能越贴合。

2. **回归误差（RMSE / MAE）**  
   - 若将互补度分数映射到相同尺度（比如先归一化到 \([0,1]\)），再与真实性能做回归对比。  
   - 定义：  
     
     ```
     RMSE = sqrt( (1/N) × sum( (y_hat(A,B) - y_true(A,B))^2 ) )
      MAE  = (1/N) × sum( |y_hat(A,B) - y_true(A,B)| )
     ```
   - RMSE/MAE 越低，说明预测越准确。
   
3. **分类准确率（当互补度二值化或分等级时）**  
   - 可以将真实性能损失离散化为“低冲突（0）”与“高冲突（1）”。  
   - 若互补度分数 \(\hat y\) 大于某阈值则判断为“低冲突”，否则“高冲突”。  
   - 计算分类准确率（Accuracy）、Precision、Recall、F1-score 等，比较不同公式在二分类任务上的性能。

4. **Top-k 冲突对 排序准确率**  
   - 离线对所有可能的线程对 \((A,B)\) 进行排序：按真实同核损失取前 \( k\% \) 的组合称为“最冲突对”。  
   - 再用预测分数排序，取前 \( k\% \) 看与真实“最冲突对”重叠的比例（Precision@k、Recall@k）。  
   - 如果你的新公式能更准确地把“最糟糕的冲突对”排在前面，就说明在调度时更可靠。

## 四、具体对比实验流程建议

下面以“两线程对照” 为主，对比流程可分为以下几步：

### 1. 挑选代表性线程对 / 场景

- **纯 CPU 密集 vs 纯 CPU 密集**  
  比如两个不同的 `sysbench --cpu` 任务或两种计算密集型程序。  
- **CPU 密集 vs 内存密集**  
  一个高 IPC 计算程序（比如 `sysbench-cpu`），一个内存带宽或缓存敏感程序（比如 `stream`、`membench`）。  
- **CPU 密集 vs I/O 密集**  
  一个计算负载（`sysbench-cpu`），一个磁盘 I/O 负载（`fio` 或 `iozone`）。  
- **生产型混合**  
  如 Nginx 或 Redis 服务与计算任务混合。

针对每对线程 \((A,B)\)：

1. **同核并行运行**：在同一个物理核上同时启动 A 和 B，记录 “并行运行后” 的真实组合性能指标（如 IPC 总和、吞吐量或执行时间）。  
2. **不同核隔离运行**：分别让 A/B 运行在不同物理核，记录隔离时的性能指标。  
3. **计算真实性能损失**：  
   
   ```
   Δ_loss(A,B) = (Perf_iso(A) + Perf_iso(B)) - Perf_same_core(A,B)
   ```
   
   将 Δ_loss 作为 “真实标签”  y_true(A,B)。

### 2. 用 eBPF 同步采集硬件事件

在“同核并行” 运行期间，使用你的两个 eBPF 程序分别采集：  
- 线程 A 在该核心一段时间窗口内的平均 L1 miss、L2 miss、iTLB miss、dTLB miss、IPC、分支未命中次数等。  
- 线程 B 同样的硬件事件统计。  
最后得到每对 \((A,B)\) 的向量特征 \(\mathbf{v}_A\) 与 \(\mathbf{v}_B\)。

### 3. 计算各方法的预测分数

- **新公式** C_new(A,B)。  
- **Baseline \#1：IPC 差公式** C_IPC(A,B)。  
- **Baseline \#2：L2 差公式** C_L2(A,B)。  
- **Baseline \#3：文献加权和** C_weighted(A,B)。  
- **Baseline \#4：方差法** C_var(A,B)。

如果需要对分数做归一化，可将所有分数缩放到 \([0,1]\) 区间，以便与真实 \(\Delta_{\text{loss}}\) 做回归对比。

### 4. 对比结果

1. **相关系数 / 回归误差**  
   - 计算 Pearson/Spearman 相关系数：  
     
     ```
     r = corr( y_hat(A,B), y_true(A,B) )
     ```
     
     相关系数越高，预测越贴合真实。  
   - 计算 RMSE / MAE：  
     
     ```
     RMSE = sqrt( (1/N) × sum( (y_hat(A,B) - y_true(A,B))^2 ) )
      MAE  = (1/N) × sum( |y_hat(A,B) - y_true(A,B)| ) 
     
     ```
     
     RMSE/MAE 越低，说明预测越准确。
   
2. **Top-k 排序准确率**  
   - 按真实 Δ_loss 排序，取前若干最冲突的组合（例如前 10%），记为集合  S_true。  
   - 按预测分数  y_hat排序，取前同样比例的组合，记为 S_pred。  
   - 计算  
     Precision@k = |S_true ∩ S_pred| / |S_pred|
      Recall@k    = |S_true ∩ S_pred| / |S_true|
   - 新公式若能更准确地把“最糟糕的冲突对”排到前面，就说明在调度时更可靠。

3. **分类准确率**  
   - 将 Δ_loss > T_th定义为“高冲突”（1），否则“低冲突”（0）。  
   - 把互补度分数  y_hat 也二值化（分数 < 阈值判为“高冲突”），计算分类准确率、F1-score 等。

4. **可视化**  
   - **散点图**：横轴为某方法预测分数  y_hat，纵轴为真实性能损失 Δ_loss，绘制趋势线，让人直观感受拟合程度。  
   - **误差柱状图／折线图**：对比不同方法在同一测试集上的 RMSE/MAE。  
   - **Top-k 曲线**：绘制 Precision@k 随 k 变化的曲线，对比各方法的表现。

### 5. （可选）多线程组初步验证

- 虽然重点是两线程对照，但也可尝试简单的“三线程”／“四线程” 情形：  
  1. 在离线环境下，选择三条线程 {T1,T2,T3}，分别测量它们一起同核运行时的整体性能指标，与分到不同核运行的情况对比，得到真实组损失 Δ_loss^(3)。  
  2. 用两两公式先算出三对互补度 C(T1,T2), C(T1,T3), C(T2,T3)，再平均得到 C_group^(A)。  
  3. 比较 C_group^(A)与真实 Δ_loss^(3) 的相关性／误差，看两两平均方案在  k=3 时是否仍然有效。  
  4. 如果效果不好，再尝试“方差方案”或“混合方案”验证。

## 五、如何在论文/报告里突出新公式优越性

1. **公式推导与直观示例**  
   - 先写出你新公式的完整形式，解释每一项（为什么要纳入 L1 miss、L2 miss、iTLB、dTLB、分支未命中、IPC 差距等）。  
   - 给一个简单数值示例：假设线程 A：{L1miss=100, L2miss=200, IPC=2.5}、线程 B：{150, 180, 1.0}，分别带入"只 IPC 差"公式和你新公式，展示两者的互补度分数有何差异，并解释为何你的结果更贴合直观。
2. **列出所有 Baseline 方法**  
   - 明确写出你所对比的已有方法（可参考上文“可选的对比对象”部分）。  
   - 若文献中没有公开权重，写明自己如何选择／调参，保证对比公平。

3. **对比表格与可视化**  
   - 放一个表格：各方法在典型两线程场景下的 Pearson r、RMSE、MAE。将最优值用 **粗体** 标出。  
   - 散点图：展示“真实损失 vs 预测分数”的拟合情况，对比曲线斜率与拟合优度。  
   - Top-k 准确率曲线：展示新公式能更准确地定位“最糟糕冲突对”。

4. **不同场景下的分析**  
   - 例如在纯 CPU 密集场景时，各方法差距较小；但在“CPU vs 内存密集”或“CPU vs I/O 密集” 时，你的公式因为纳入更多硬件事件维度，表现明显优于只看 IPC 差值的方式。  
   - 用定性叙述 & 定量数据结合，让读者直观感受到“为何多维度综合优于单一维度”。

## 六、**公式**
竞争度(r, S) = f(组内使用方差, 共享资源压力)
瓶颈互补度(S) = g(瓶颈类型分布, 瓶颈强度)

架构因子 = h(缓存层级, 内存通道, 核心拓扑)

C(S) = α×资源竞争度 + β×瓶颈互补度 + γ×架构因子

从两两到群体：不是简单平均两两互补度，而是直接计算组属性

瓶颈导向：重点关注造成性能下降的关键资源

架构感知：考虑具体硬件拓扑的影响

动态适应性：能适应工作负载阶段变化

先从瓶颈分类+资源竞争的简单组合开始

确保理论合理性和可解释性

## 小结

- **必须做对比实验**，才能证明新公式优越：至少选 2～3 个 Baseline 方法（只 IPC 差、只 L2 差、文献加权和、方差法），在同一批实验数据上运行，比较 Pearson correlation、RMSE/MAE、Top-k 准确率等指标。  
- **保持实验环境与统计方法统一**，比如所有方法在同一硬件、同一负载组合下测试，且超参数都严格调优，保证公平性。  
- **突出论文创新点**：  
  1. 在多个场景下，新公式的相关系数更高、误差更小；  
  2. Top-k 在定位“最冲突对”时更精准；  
  3. 用可视化图表（散点、误差柱状、Top-k 曲线）辅助说明。  

通过以上对比与分析，你可以充分证明：「引入更多硬件事件维度并合理加权」这一新公式，能够更准确地刻画线程间真实的互补性，从而在调度决策中取得更好效果。
